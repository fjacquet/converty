<task type="auto">
  <name>Install Workbox dependencies and create service worker</name>
  <files>
    package.json
    public/sw.js
  </files>
  <action>
Install Workbox runtime libraries for service worker implementation:

```bash
npm install workbox-window
npm install -D workbox-build
```

DO NOT install workbox-precaching, workbox-routing, workbox-strategies separately - these will be loaded via importScripts in the service worker from CDN (Workbox v7 standard pattern).

Create public/sw.js implementing:

1. Import Workbox from CDN using importScripts:
   ```javascript
   importScripts('https://storage.googleapis.com/workbox-cdn/releases/7.0.0/workbox-sw.js');
   ```

2. Configure Workbox with skipWaiting and clientsClaim for immediate activation

3. Implement caching strategies per resource type (from RESEARCH.md patterns):
   - HTML/Document requests: NetworkFirst with 7-day cache fallback (calculators work offline)
   - Static assets (JS, CSS, images): CacheFirst with 30-day expiration (long-lived, immutable)
   - Fonts: StaleWhileRevalidate with 20 entry limit (balanced freshness)

4. Add cache names: 'pages-cache-v1', 'static-cache-v1', 'font-cache-v1'

5. Add CacheableResponsePlugin (cache only status 0, 200)

6. Add ExpirationPlugin with maxEntries and maxAgeSeconds per strategy

7. Add offline fallback: If document fetch fails and not in cache, serve simple offline message

WHY Workbox CDN: Research shows importScripts pattern is standard for Workbox v7, avoids bundling issues with static export, and provides automatic cache versioning.

WHY these strategies: NetworkFirst for HTML ensures fresh content when online but offline fallback. CacheFirst for static assets leverages immutability (Next.js content hashes). Research confirms this pattern for static site PWAs.

IMPORTANT: Do NOT use precacheAndRoute with self.__WB_MANIFEST yet - that requires build integration (Plan 03). Use runtime caching only for now.
  </action>

   - CacheFirst, NetworkFirst, StaleWhileRevalidate strategies
3. npx tsc --noEmit passes (no TypeScript errors)
4. Service worker syntax valid: Open public/sw.js in VS Code, check for syntax errors

<task type="auto">
  <name>Create offline detection hook and banner component</name>
  <files>
    src/lib/pwa/offline-detector.ts
    src/components/ui/offline-banner.tsx
  </files>
  <action>
Create offline detection infrastructure following RESEARCH.md Pattern 4.

1. Create src/lib/pwa/offline-detector.ts:
   - Export useOnlineStatus() hook (client-side only)
   - useState(true) for isOnline (default online)
   - useEffect to:
     - Initialize: setIsOnline(navigator.onLine)
     - Add event listeners: window.addEventListener('online', handleOnline) and window.addEventListener('offline', handleOffline)
     - Cleanup: removeEventListener in return function
   - Return isOnline boolean

2. Create src/components/ui/offline-banner.tsx:
   - "use client" directive (client component)
   - Import useOnlineStatus from offline-detector
   - If isOnline, return null (no banner)
   - If offline, render fixed banner at top with:
     - Tailwind classes: "fixed top-0 left-0 right-0 z-50 bg-yellow-500 text-black px-4 py-2 text-center text-sm font-medium"
     - Message: "You are offline. Calculators will work from cache."
     - Icon: Optional - use lucide-react WifiOff icon if desired

3. Export OfflineBanner component for use in layout (Plan 03 will add to layout)

WHY navigator.onLine + events: Research shows navigator.onLine alone can be stale. Event listeners ('online', 'offline') provide real-time updates when network status changes.

WHY fixed banner: Non-intrusive, always visible, auto-dismisses when back online. Yellow (warning) color is standard for transient states.

IMPORTANT: Both files must have "use client" since they use React hooks and browser APIs not available server-side.
  </action>
