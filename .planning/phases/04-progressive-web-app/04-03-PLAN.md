  <action>
Create service worker registration infrastructure following RESEARCH.md Pattern 1.

1. Create src/lib/pwa/register-sw.ts:
   - Export registerServiceWorker() function
   - Check if 'serviceWorker' in navigator (browser support)
   - Check process.env.NODE_ENV === 'production' (dev mode check)
   - If both true, call navigator.serviceWorker.register('/sw.js', { scope: '/' })
   - Handle registration promise:
     - Success: console.log('Service worker registered:', registration.scope)
     - Error: console.error('Service worker registration failed:', error)
   - Return void (fire-and-forget pattern)
   - Add TypeScript types for all parameters and return values

2. Update src/app/[locale]/layout.tsx to add registration:
   - Create new client component: src/app/[locale]/sw-registration.tsx
   - "use client" directive
   - Import useEffect from react
   - Import registerServiceWorker from @/lib/pwa/register-sw
   - Import OfflineBanner from @/components/ui/offline-banner
   - Import InstallPrompt from @/components/ui/install-prompt
   - useEffect(() => { registerServiceWorker(); }, []) for one-time registration
   - Render OfflineBanner component (from Plan 02)
   - Render InstallPrompt component (from Task 3)
   - Return null for the registration effect itself (registration happens in effect, no UI)

3. In src/app/[locale]/layout.tsx:
   - Import SWRegistration component
   - Add <SWRegistration /> INSIDE the ThemeProvider but BEFORE the main div
   - This ensures SW registers on first page load regardless of route

WHY production-only registration: Research (RESEARCH.md Pitfall 5) shows service worker caching breaks hot reload in development. Development needs fresh code on every change.

WHY separate client component: Root layout is server component. Service worker registration requires browser APIs (navigator, useEffect). Separate client component provides clean boundary.

WHY scope '/' : Converty uses locale routing (/en/, /fr/, etc). Service worker at root scope intercepts all requests. If scope was /en/, would miss /fr/ requests.

WHY integrate InstallPrompt here: Task 3 creates the component, but it needs to be rendered in the layout to be visible to users. Including it in SWRegistration component ensures it's available globally.

IMPORTANT: Do NOT add to root layout (src/app/layout.tsx) - that's just HTML shell. Add to locale layout where application logic lives.

<task type="auto">
  <name>Create build integration script for service worker</name>
  <files>
    scripts/generate-sw.js
    package.json
  </files>
  <action>
Create build-time script to generate service worker with precache manifest using Workbox CLI.

1. Create scripts/generate-sw.js:
   - Import workbox-build: const { generateSW } = require('workbox-build');
   - Configure generateSW with:
     - globDirectory: 'out' (Next.js static export output)
     - globPatterns: ['**/*.{html,js,css,png,jpg,jpeg,svg,webp,woff2}']
     - swDest: 'out/sw.js' (output location)
     - skipWaiting: true (activate immediately)
     - clientsClaim: true (take control of pages)
     - runtimeCaching array:
       - HTML documents: NetworkFirst with pages-cache, 7-day expiration
       - Images: CacheFirst with images-cache, 30-day expiration, 100 max entries
       - Fonts: StaleWhileRevalidate with font-cache, 20 max entries
   - Call generateSW and log result: `Generated SW, precaching ${count} files (${size} bytes)`
   - Add error handling: catch and exit with code 1 on failure

2. Update package.json scripts:
   - Rename current "build" to "build:next": "next build"
   - Create new "build": "next build && node scripts/generate-sw.js"
   - This ensures SW generation happens AFTER Next.js build completes

3. Add workbox-build validation:
   - Script should check workbox-build is installed (already added in Plan 02)
   - If missing, provide clear error: "workbox-build not found. Run: npm install -D workbox-build"

WHY post-build script: Research shows workbox-build needs static files to exist before generating precache manifest. Next.js build creates files in out/, then script scans and injects manifest.

WHY generateSW not injectManifest: public/sw.js from Plan 02 uses Workbox CDN runtime. generateSW creates complete SW file with precaching. We'll REPLACE public/sw.js with generated version during build.

IMPORTANT: This means public/sw.js serves as template/reference. Build process generates actual out/sw.js that production uses. This is standard pattern for static exports.

NOTE: After this task, public/sw.js can be simplified to just Workbox CDN import since generateSW handles everything. Or keep it for development reference.
  </action>

- Import lucide-react icons: Download, Share

Define interface for beforeinstallprompt event:
```typescript
interface BeforeInstallPromptEvent extends Event {
  prompt: () => Promise<void>;
  userChoice: Promise<{ outcome: 'accepted' | 'dismissed' }>;
}
```

Component state:
- installPrompt: BeforeInstallPromptEvent | null
- isIOS: boolean (detect iOS devices)
- isStandalone: boolean (detect if already installed)

useEffect for initialization:
1. Detect iOS: /iPad|iPhone|iPod/.test(navigator.userAgent)
2. Detect standalone mode: window.matchMedia('(display-mode: standalone)').matches
3. Add beforeinstallprompt listener (Android/Desktop Chrome only)
4. Prevent default on event, save to state
5. Cleanup: remove listener on unmount

Handle install button click:
- Call installPrompt.prompt()
- Await installPrompt.userChoice
- If accepted, clear prompt state (hide button)

Render logic:
1. If already installed (isStandalone), return null
2. If iOS and no install prompt: Show iOS instructions with Share icon
   - Message: "Install this app: tap Share then 'Add to Home Screen'"
   - Use info box styling (blue background)
3. If install prompt available: Show install button
   - Button text: "Install Converty"
   - Include Download icon
   - Primary button styling

Styling:
- Use Tailwind classes matching existing button/card components
- iOS instructions: Subtle info box, not intrusive
- Install button: Primary CTA button (prominent but not annoying)
- Fixed position bottom-right or bottom-center (non-blocking)

WHY feature detection: Research (Pitfall 4) shows iOS doesn't support beforeinstallprompt. Must detect platform and show appropriate UI.

WHY standalone check: Users who already installed shouldn't see install prompt. Display mode detection is reliable check.

IMPORTANT: Progressive enhancement - component gracefully handles all scenarios: iOS, Android, Desktop, already installed, not installable. Never breaks or shows broken UI.

NOTE: This component will be integrated into the layout in Task 1 via the SWRegistration component.
