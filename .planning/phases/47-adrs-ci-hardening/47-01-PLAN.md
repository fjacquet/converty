**Header:**
```
# ADR-013: Error Boundaries and Toast Notifications
**Status:** Accepted
**Date:** 2026-02-26
**Proposed by:** v7.0 Framework Migration
```

**Context section** must cover:
- Before Phase 42, calculator crashes (unhandled exceptions) produced blank white screens with zero user feedback
- The `ConverterLayout` component had no error boundary — any runtime exception in a calculator unmounted the entire page silently
- Copy-to-clipboard and CSV/PDF export operations had no success/failure feedback
- Calculation errors (invalid inputs producing null results) were silent

**Decision section** must cover:
- Adopted `react-error-boundary` (wrapping ConverterLayout) for graceful crash recovery
- `CalculatorErrorBoundary` component with `CalculatorErrorFallback` providing a "Reload Calculator" button
- `sanitize.ts` uses `isomorphic-dompurify` (not plain `dompurify`) to prevent `window is not defined` during static generation
- Adopted `sonner` for toast notifications; `<Toaster>` mounted inside ThemeProvider in root layout
- Toast i18n keys stored under `common.toast` namespace (7 keys: copy success/fail, CSV/PDF export success/fail, calculation error)
- `onCalculationError` callback in `createCalculatorStore` is opt-in (not default) to prevent toast spam across all 169 calculators that return null for normal incomplete-input states
- Callback signature: `(values: T) => string` — typed, no any, caller decides message text

**Alternatives Considered:**
- React's built-in error handling (no fallback UI, no recovery path)
- react-toastify (more complex API, heavier bundle; sonner has simpler API)
- Plain DOMPurify (causes `window is not defined` during Next.js static generation — rejected)

**Consequences:**
Positive:
- Calculator crashes now show a graceful fallback with reload button
- Users receive clear feedback on copy/export operations
- `sanitize.ts` utility sanitizes any HTML rendered from user input or external data
- Error boundary is server-component-compatible — ConverterLayout remains a server component
Negative/Tradeoffs:
- `onCalculationError` requires explicit opt-in per calculator (extra boilerplate for complex calculators)
- ~2 additional packages added to bundle

**Implementation Details:**
- `src/components/error-boundary/calculator-error-boundary.tsx` — uses `'use client'` (React state internally)
- `src/components/error-boundary/calculator-error-fallback.tsx` — uses `'use client'`
- `src/lib/sanitize.ts` — isomorphic-dompurify wrapper
- `src/hooks/use-copy-to-clipboard.ts` — hook-level toast covers all hook consumers automatically
- Toaster placed inside ThemeProvider after flex container div in root layout

**Related ADRs:** ADR-007 (pure functions), ADR-009 (no backend / privacy first)

Create `docs/adr/ADR-014-zod-input-validation.md` documenting the Phase 43 decision to adopt Zod for runtime input validation.

**Header:**
```
# ADR-014: Zod Runtime Input Validation
**Status:** Accepted
**Date:** 2026-02-26
**Proposed by:** v7.0 Framework Migration
```

**Context section** must cover:
- All 169 calculator inputs were validated only by TypeScript at compile time — runtime validation was absent
- URL parameter parsing used custom `parseNumberParam`/`parseBooleanParam`/`parseStringParam` helpers that produced no structured errors
- Invalid inputs produced silent null results with no user feedback
- The `createCalculatorStore` factory had a `validate` callback but no structured schema support

**Decision section** must cover:
- Installed `zod` as a runtime dependency (not devDep) — schemas execute client-side in stores
- Zod schemas defined for each calculator's input type in `src/lib/schemas/`
- `createCalculatorStore` updated to accept optional `schema?: ZodType` parameter; explicit `validate` takes precedence over `schema` when both provided
- `ZodType` imported as type-only (`import type`) for Biome strict compliance; `z` imported as value in url-params.ts
- String-based schemas (`z.string().refine()`) used for health/math categories — `z.coerce.number()` was rejected because it outputs `number` type incompatible with string-typed `FormValues`
- Finance calculators use `z.number()` since their `FormValues` are number types
- Corpulence calculator uses `useState` directly — schema created but store wiring skipped (no `createCalculatorStore` to attach to)
- Only 4 of 23 finance calculators wired with schema (compound-interest, loan, mortgage, retirement) — only ones using `createCalculatorStore`
- Photo/video/automotive/cooking/network/crypto/web/physics/music/color: schemas created for reference only; datetime (9) and data (2) calculators fully wired
- Zod `.safeParse()` replaces custom URL param helpers; backward-compatible with existing URL state

**Alternatives Considered:**
- Yup (similar API but larger bundle; Zod is TypeScript-native, ships its own types)
- valibot (newer, smaller but less mature ecosystem in 2026)
- Custom validation functions only (already existed as `validate:` callback; Zod adds schema-first approach with better error messages)

**Consequences:**
Positive:
- Runtime validation catches out-of-range inputs before calculation runs
- Field-level error messages surface to users (not silent null results)
- URL parameters validated via `.safeParse()` with structured error reporting
- Schema-first approach documents expected input shapes explicitly

Negative/Tradeoffs:
- ~14KB gzipped bundle addition (zod runtime)
- Not all calculators fully wired — useState-based calculators have schemas but no store integration
- Two validation paths coexist: `schema` (Zod) and `validate` (legacy callback)

**Implementation Details:**
- `src/lib/schemas/` — one schema file per category
- `src/lib/stores/calculator-store.ts` — `schema?: ZodType` added to config
- `src/lib/utils/url-params.ts` — Zod `.safeParse()` replaces custom helpers

**Related ADRs:** ADR-002 (URL state sync), ADR-003 (Zustand state management)

**Header:**
```
# ADR-015: LZ-String URL State Compression
**Status:** Accepted
**Date:** 2026-02-26
**Proposed by:** v7.0 Framework Migration
```

**Context section** must cover:
- Complex calculators (subnet with 8+ params, CPU comparison with 4 CPU IDs + filters) produced very long URLs with per-key search params
- Sharing these URLs via messaging apps, email, or QR codes was impractical due to length
- The existing url-sync middleware wrote one `URLSearchParams` key per form field — a CPU comparison URL could exceed 300 characters
- A dead-code bug existed: `Object.keys(urlParams)` called on a `URLSearchParams` Map always returns `[]` — the block was never executing

**Decision section** must cover:
- Installed `lz-string@1.5.0` — ships its own TypeScript typings, no `@types/lz-string` needed
- `compressToEncodedURIComponent` produces URL-safe base64 output (no `%`-encoding of compression characters)
- Write path: serialize full state to JSON, compress, write as single `?z=` search param
- Read path dual-path: `?z=` (new compressed) takes priority; fallback to legacy per-key params (`urlParams.size > 0`) for backward compatibility
- Null-safety guard required before `JSON.parse` — `decompressFromEncodedURIComponent` returns `null` on invalid input
- Prototype pollution prevention: only keys present in `initialValues` are accepted from parsed JSON
- Fixed `Object.keys(urlParams)` Map bug (dead code removed) discovered during this work
- Search params used (not hash `#`) — maintains GitHub Pages compatibility (ADR-008)

**Alternatives Considered:**
- Hash-based compression (rejected: incompatible with GitHub Pages static routing — see ADR-008)
- pako/zlib compression (produces binary output requiring base64 encoding; lz-string produces URL-safe output directly)
- No compression (accepted status quo for simple calculators; too long for complex ones)
- LZString via hash with custom routing (rejected: adds routing complexity)

**Consequences:**
Positive:
- URL length reduced 60-80% for complex calculators (CPU comparison: ~320 chars → ~85 chars)
- Single `?z=` param makes URLs cleaner and easier to share
- Backward-compatible: pre-compression URLs (without `?z=`) continue to work
- Round-trip verified: compress → decompress is lossless (tests in Phase 44)
- Prototype pollution prevented by key allowlist from `initialValues`

Negative/Tradeoffs:
- Compressed URLs are opaque (cannot be read or edited manually)
- Users cannot construct a URL by hand for a specific calculator state
- ~3KB gzipped bundle addition

**Implementation Details:**
- `src/lib/stores/url-sync.ts` — write path updated to use `compressToEncodedURIComponent`
- `src/lib/stores/calculator-store.ts` — read path: check `?z=` first, fall back to legacy params
- Tests verify round-trip losslessness in `src/lib/stores/__tests__/url-sync.test.ts`

**Related ADRs:** ADR-002 (URL state sync — extended by this ADR), ADR-008 (GitHub Pages deployment — constraint on hash routing)
