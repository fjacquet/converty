## 1. VMware vSphere Storage Capacity Planning

### Core Calculation Components

**VM Disk Space:**
- **Total Provisioned**: Sum of all VM provisioned disk capacity
- **Expected Used**: Derived from total provisioned for storage over-subscription percentage
- **Over-subscribed Capacity**: Difference between provisioned and used (thin provisioning)

**Additional Storage:**
- **RAM**: Memory capacity allocated to VM (for swap file calculation)
- **Snapshot Allocation**: Additional storage for VM snapshots (% of provisioned)
- **Storage Over-subscription**: Expected thin provisioning level (0% = no thin provisioning)

**vSAN-Specific:**
- Formula: `Subscription capacity = Total TiBs × Number of ESXi hosts in vSAN cluster`

### Implementation Formula

```typescript
// VM Storage Calculator inputs
interface VMStorageInputs {
  vmCount: number;
  diskSizePerVM: number; // GB
  ramPerVM: number; // GB (for swap files)
  snapshotAllocation: number; // percentage
  overSubscription: number; // percentage (thin provisioning)
  redundancyFactor: number; // RAID level overhead
}

// Calculations
totalProvisioned = vmCount × diskSizePerVM
swapSpace = vmCount × ramPerVM
snapshotSpace = totalProvisioned × (snapshotAllocation / 100)
usedSpace = totalProvisioned × (1 - overSubscription / 100)
totalRequired = usedSpace + swapSpace + snapshotSpace
totalWithRedundancy = totalRequired × redundancyFactor
```

## 2. Kubernetes Cluster Capacity Planning

### Current Industry Statistics (2026)

- **Average CPU utilization**: Only 13% of requested CPU actually used
- **Resource usage**: 20-45% of requested resources power actual workloads
- **Waste estimate**: Up to 30% of cloud spending wasted on over-provisioning (BCG)

### Resource Request Best Practices

**CPU and Memory Requests:**
- Set requests between **85-115% of average usage**
- Pods scheduled based on requests only (not limits)
- Node capacity checked: `node_free_capacity ≥ pod_cpu_request + pod_memory_request`

**Utilization Targets:**
- **Target**: 70% utilization for CPU and memory
- **Never**: 100% utilization (no headroom for spikes)

### Key Metrics for Calculation

**Node Metrics:**
- `node_cpu_seconds_total`
- `node_memory_MemTotal_bytes`
- `node_memory_MemAvailable_bytes`

**Pod Metrics:**
- `kube_pod_container_resource_requests`
- `container_cpu_usage_seconds_total`
- `container_memory_working_set_bytes`
- `kube_pod_container_resource_limits`

### Implementation Formula

```typescript
// Kubernetes Capacity Calculator inputs
interface K8sCapacityInputs {
  // Workload
  podCount: number;
  cpuPerPod: number; // millicores
  memoryPerPod: number; // Mi

  // Node specs
  cpuPerNode: number; // cores
  memoryPerNode: number; // GB

  // Overhead
  systemReserved: number; // percentage (typically 20%)
  targetUtilization: number; // percentage (typically 70%)
}

// Calculations
totalCpuRequest = podCount × cpuPerPod / 1000 // convert to cores
totalMemoryRequest = podCount × memoryPerPod / 1024 // convert to GB

usableCpuPerNode = cpuPerNode × (1 - systemReserved / 100)
usableMemoryPerNode = memoryPerNode × (1 - systemReserved / 100)

// Nodes needed (account for target utilization)
nodesByCpu = Math.ceil(totalCpuRequest / (usableCpuPerNode × targetUtilization / 100))
nodesByMemory = Math.ceil(totalMemoryRequest / (usableMemoryPerNode × targetUtilization / 100))
nodesRequired = Math.max(nodesByCpu, nodesByMemory)
```

```typescript
// VMware Licensing Calculator inputs
interface VMwareLicensingInputs {
  hostCount: number;
  cpuPerHost: number;
  coresPerCpu: number;
```

## 4. Server Virtualization Ratio (ESX Host Sizing)

### Virtualization Best Practices

**vCPU-to-Core Ratios:**
- **VDI workloads**: 7-11 desktops per core (ideal)
- **Maximum supported**: 16 VMs per core (ESXi limit)
- **CPU-intensive workloads**: Use lower ratios

**Memory Planning:**
- **ESXi overhead**: 1024 MB (1 GB) reserved for hypervisor
- **Management buffer**: 20% reserved for vMotion, HA, management
- **Effective memory**: `Total RAM - 1 GB - 20%`

**Utilization Targets:**
- **CPU**: Target 70% utilization
- **Memory**: Target 70% utilization
- **Never**: 100% utilization (no headroom)

**High Availability (N+1):**
- Cluster must survive 1 host failure
- Required hosts: `Minimum hosts + 1`

### Implementation Formula

```typescript
// Server Virtualization Calculator inputs
interface ESXHostSizingInputs {
  // VM Workload
  vmCount: number;
  vCpuPerVM: number;
  ramPerVM: number; // GB

  // Host Specifications
  coresPerHost: number;
  ramPerHost: number; // GB

  // Constraints
  vCpuToCorRatio: number; // default: 4 for general workloads
  targetUtilization: number; // default: 70%
  highAvailability: boolean; // N+1 redundancy
}

// Calculations
totalVCpus = vmCount × vCpuPerVM
totalRamRequired = vmCount × ramPerVM

// Host capacity
usableCoresPerHost = coresPerHost // logical cores (with hyperthreading)
effectiveVCpuPerHost = usableCoresPerHost × vCpuToCorRatio
effectiveRamPerHost = (ramPerHost - 1) × 0.8 // 1GB hypervisor, 20% buffer

// Hosts needed (account for target utilization)
hostsByCpu = Math.ceil(totalVCpus / (effectiveVCpuPerHost × targetUtilization / 100))
hostsByMemory = Math.ceil(totalRamRequired / (effectiveRamPerHost × targetUtilization / 100))
hostsRequired = Math.max(hostsByCpu, hostsByMemory)

// Add HA redundancy
if (highAvailability) {
  hostsRequired = hostsRequired + 1 // N+1
}
```

## 5. Virtualization Cost Calculator

### Cost Components

**Infrastructure Costs:**
1. **Hardware**
   - Server hardware (CPU, RAM, storage, networking)
   - Licensing (VMware, Windows Server, etc.)
   - Networking equipment (switches, NICs)
   - Storage arrays (SAN, NAS)

2. **Software**
   - Hypervisor licensing (VCF, VVF, vSphere)
   - Management tools (vCenter, Aria Suite)
   - Backup software
   - Monitoring tools

3. **Operational**
   - Power consumption ($/kWh × Watts × hours)
   - Cooling costs
   - Data center space ($/rack unit)
   - Support contracts
   - Personnel (admin hours)

### Implementation Formula

```typescript
// Virtualization Cost Calculator inputs
interface VirtualizationCostInputs {
  // Hardware
  serverCost: number; // per server
  serverCount: number;
  storageCost: number; // total SAN/NAS cost
  networkCost: number; // switches, cables

  // Software (annual)
  vmwareLicenseCost: number; // from licensing calculator
  osLicenseCost: number; // Windows Server, etc.
  backupLicenseCost: number;

  // Operational (annual)
  powerCostPerKwh: number; // $/kWh
  wattsPerServer: number;
  coolingMultiplier: number; // typically 1.3-1.5 (PUE)
  datacenterCostPerRU: number; // $/rack unit/year
  rackUnitsPerServer: number;
  adminHoursPerYear: number;
  costPerAdminHour: number;

  // Term
  termYears: number;
}

// Calculations
// Capital Expenditure (one-time)
capex = (serverCost × serverCount) + storageCost + networkCost

// Operational Expenditure (annual)
powerAnnual = (wattsPerServer × serverCount / 1000) × 24 × 365 × powerCostPerKwh × coolingMultiplier
softwareAnnual = vmwareLicenseCost + osLicenseCost + backupLicenseCost
datacenterAnnual = datacenterCostPerRU × rackUnitsPerServer × serverCount
laborAnnual = adminHoursPerYear × costPerAdminHour

opexAnnual = powerAnnual + softwareAnnual + datacenterAnnual + laborAnnual
opexTotal = opexAnnual × termYears

// Total Cost of Ownership
tco = capex + opexTotal

// Cost per VM
costPerVM = tco / (vmCount × termYears)
```

## 6. CodeQL Security Remediation

### Remote Property Injection (High Severity)

**Vulnerability:**
CodeQL detects dynamically computing object property names from untrusted input, which can allow attackers to overwrite vital properties like `__proto__`. This is **prototype pollution**.

**Attack Vector:**
```typescript
// VULNERABLE CODE
function setUrlParam(obj: any, key: string, value: string) {
  obj[key] = value; // key comes from URL params - DANGEROUS!
}

// Attacker can inject: ?__proto__[isAdmin]=true
```

**Remediation Strategies:**

1. **Whitelist Approach** (Recommended):
```typescript
const ALLOWED_PARAMS = new Set(['page', 'filter', 'sort']);

function setUrlParam(obj: Record<string, string>, key: string, value: string) {
  if (ALLOWED_PARAMS.has(key)) {
    obj[key] = value;
  }
}
```

2. **Object.create(null)** (No prototype):
```typescript
const params = Object.create(null); // no __proto__
params[key] = value; // safe, no prototype to pollute
```

3. **hasOwnProperty Check**:
```typescript
if (Object.prototype.hasOwnProperty.call(obj, key) && !key.includes('__proto__')) {
  obj[key] = value;
}
```

4. **Use Map Instead of Objects**:
```typescript
const params = new Map<string, string>();
params.set(key, value); // Map doesn't have prototype pollution
```

**For Converty's url-params.ts:**
- Current code uses dynamic property access from URL
- Fix: Whitelist known calculator parameters
- Alternative: Use Map for URL state instead of plain objects

**Sources:**
- [CodeQL Remote Property Injection Documentation](https://codeql.github.com/codeql-query-help/javascript/js-remote-property-injection/)
- [CodeQL Code Injection Guide](https://codeql.github.com/codeql-query-help/javascript/js-code-injection/)

### Unused Variables/Imports (Note Severity)

**Issue:**
- Unused imports and variables increase bundle size
- Can confuse developers (dead code)
- May indicate incomplete refactoring

**Fix Strategy:**
```typescript

// AFTER
```
