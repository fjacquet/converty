<objective>
Add an opt-in `onCalculationError` callback to `createCalculatorStore`. When provided, it fires `toast.error()` inside `setValue`/`setValues` only when the user has changed a value and `calculate()` returns null. The opt-in design prevents toast spam on calculators where null is a normal "not enough input yet" state.

Purpose: Satisfies R2.5 without polluting all 169 calculators. Per research recommendation: opt-in, no default behavior change.
Output: 1 modified file (calculator-store.ts). Existing stores unaffected. New `onCalculationError` option available for adoption.

  <action>
    Update `src/stores/calculator-store.ts` to support an optional `onCalculationError` callback.

    **Step 1: Add import**
    Add `import { toast } from "sonner";` at the top of the file, after the existing imports.

    **Step 2: Extend CreateCalculatorStoreOptions interface**
    Add the optional field to `CreateCalculatorStoreOptions<T, R>`:
    ```typescript
    /**
     * Optional callback invoked when calculate() returns null after a user action.
     * Return a string message to show as a toast.error notification.
     * NOT called on initial mount — only called from setValue/setValues.
     */
    onCalculationError?: (values: T) => string;
    ```

    **Step 3: Destructure the new option in the factory function**
    Update the destructuring in `createCalculatorStore`:
    ```typescript
    export function createCalculatorStore<T extends object, R>({
      initialValues,
      calculate,
      validate,
      syncUrl = true,
      debounceMs = 150,
      onCalculationError,
    }: CreateCalculatorStoreOptions<T, R>) {
    ```

    **Step 4: Fire toast in setValue and setValues when result is null**
    In the `setValue` action:
    ```typescript
    setValue: <K extends keyof T>(key: K, value: T[K]) => {
      const currentState = get();
      const newValues = { ...currentState.values, [key]: value };
      const errors = validate?.(newValues) ?? {};
      const result = Object.keys(errors).length === 0 ? calculate(newValues) : null;

      // Fire toast only when user changes a value and calculation fails
      if (result === null && onCalculationError) {
        toast.error(onCalculationError(newValues));
      }

      set({ values: newValues, errors, result });
    },
    ```

    In the `setValues` action:
    ```typescript
    setValues: (values: T) => {
      const errors = validate?.(values) ?? {};
      const result = Object.keys(errors).length === 0 ? calculate(values) : null;

      // Fire toast only when user changes values and calculation fails
      if (result === null && onCalculationError) {
        toast.error(onCalculationError(values));
      }

      set({ values, errors, result });
    },
    ```

    **Step 5: Do NOT add toast to the initial state creator block**
    The initial `values: mergedInitialValues, result: null` at store creation must NOT fire the callback — that would toast on page load.

    The `reset` action also must NOT call `onCalculationError` — reset is a deliberate action that sets result to null intentionally.

    Biome compliance: `onCalculationError` is called with typed `values: T` — no `any` or unknown narrowing needed.
